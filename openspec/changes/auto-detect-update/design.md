## 上下文

当前 `AchievementCrawler.get_achievement_data()` 方法（`core/crawl_tab.py:63-105`）的缓存逻辑为：缓存文件存在则直接返回缓存数据，否则请求 API 并保存缓存。该逻辑没有任何过期判断。

API 响应（`POST api.kurobbs.com/wiki/core/catalogue/item/getEntryDetail`）的顶层 `data` 对象中已包含 `lastUpdateTime` 字段（日期格式如 `2026-03-19`），可直接用于新鲜度判断。

现有"清除缓存"按钮保持不变，作为用户强制刷新的手段。

## 目标 / 非目标

**目标：**

- 在用户点击"开始爬取"时，自动检测远端数据是否有更新，有更新则刷新缓存
- 网络异常或认证缺失时优雅降级，不阻塞用户正常使用
- 兼容历史版本生成的无元数据缓存文件

**非目标：**

- 不实现定时轮询（仅在启动时自动检查一次）
- 不改变 API endpoint 或请求参数
- 不修改"清除缓存"按钮的现有行为

## 决策

### 决策 1：复用同一 API 请求获取更新时间

**选择**：发送一次与完整爬取相同的 API 请求，先解析 `lastUpdateTime` 进行比对，若数据已更新则直接使用本次请求的完整响应数据。

**替代方案**：
- 发送两次请求（一次轻量检查 + 一次完整拉取）→ 增加不必要的网络开销
- 使用 HTTP HEAD 请求 → 该 API 不支持 HEAD 方法

**理由**：API 不提供轻量的"仅返回元数据"接口，无法避免拉取完整数据。但可以先从完整响应中提取 `lastUpdateTime` 与本地比对，若未更新则丢弃响应使用缓存，若已更新则直接使用本次响应。这样始终只需一次请求。

### 决策 2：元数据存储在缓存文件顶层

**选择**：在 `achievement_cache.json` 中添加 `_cache_meta` 顶层字段。

```json
{
  "_cache_meta": {
    "lastUpdateTime": "2026-03-19",
    "cached_at": "2026-03-20T10:30:00"
  },
  "code": 200,
  "data": { ... }
}
```

**替代方案**：
- 使用独立文件（如 `cache_meta.json`）→ 增加文件管理复杂度，可能出现不一致
- 存储在 `config.json` 中 → 职责不清，config 应只管配置

**理由**：元数据与缓存数据生命周期一致（缓存删除时元数据也应删除），存在同一文件中保证原子性。`_cache_meta` 前缀下划线表示这是内部元数据，不会与 API 响应字段冲突。

### 决策 3：日期字符串直接比较

**选择**：将 `lastUpdateTime` 作为 `YYYY-MM-DD` 格式的字符串直接比较（字典序等价于日期序）。

**替代方案**：
- 解析为 `datetime` 对象比较 → 增加复杂度，无额外收益

**理由**：API 返回的 `lastUpdateTime` 格式固定为 `YYYY-MM-DD`，该格式的字典序天然等价于时间顺序，无需额外解析。

### 决策 4：修改 `get_achievement_data()` 方法的流程

**新流程**：

```
get_achievement_data()
    │
    ├── 缓存文件不存在或无 _cache_meta？
    │       → 请求 API → 保存缓存（含 _cache_meta）→ 返回数据
    │
    ├── 有缓存且有 _cache_meta
    │       → 请求 API 获取最新响应
    │       │
    │       ├── 请求失败？
    │       │       → 降级使用本地缓存 → 返回缓存数据
    │       │
    │       ├── 远端 lastUpdateTime == 本地？
    │       │       → 使用本地缓存 → 返回缓存数据
    │       │
    │       └── 远端 lastUpdateTime > 本地？
    │               → 保存新响应为缓存（含更新的 _cache_meta）→ 返回新数据
    │
    └── 认证信息缺失？
            → 有缓存则使用缓存，无缓存则报错
```

### 决策 5：启动时后台自动同步成就数据

**选择**：主窗口启动 5 秒后，通过 `QTimer.singleShot` 触发后台线程，解析远端全量成就数据并与本地 `base_achievements.json` 对比，自动合并新增、移除已删除的成就。

**替代方案**：
- 仅在用户手动点击"爬取"时同步 → 用户可能长期使用过期数据而不自知
- 启动时弹窗提示用户确认 → 增加操作步骤，降低体验

**理由**：启动静默检查对用户无干扰，后台线程不阻塞 UI，延迟 5 秒避免与启动初始化竞争资源。

### 决策 6：持久化顺序——先保存再编码

**选择**：合并成就后严格按 `save_to_json()` → `reencode_all_user_progress()` → `load_local_data()` 顺序执行。

**替代方案**：
- 先编码再保存 → `reencode_all_user_progress()` 从文件加载数据，会读到旧的成就列表，导致编码结果与实际不一致

**理由**：`reencode_all_user_progress()` 的实现依赖于从 `base_achievements.json` 文件读取成就数据，因此必须先将合并结果写入文件，再执行编码。这是开发过程中发现的关键 bug，先编码后保存会导致新成就的用户状态丢失。

### 决策 7：两层缓存校验——时间戳 + 内容 hash

**选择**：先比对 `lastUpdateTime` 字符串，时间戳相同时再比对 API 响应数据的 MD5 hash（排除 `_cache_meta` 字段），确保内容级别的一致性。

**替代方案**：
- 仅比对时间戳 → 远端可能在同一天多次更新但时间戳不变，导致遗漏更新

**理由**：双重校验在几乎不增加开销的情况下（hash 计算是本地操作）显著提高了更新检测的可靠性。`_cache_meta` 中额外存储 `data_hash` 字段用于比对。

### 决策 8：按 (名称, 去标点描述) 作为成就唯一键

**选择**：使用 `(名称, clean_desc(描述))` 二元组作为成就的唯一标识，其中 `clean_desc` 去除描述末尾的标点符号。

**替代方案**：
- 仅用名称 → 存在同名不同描述的成就（如不同版本的同名成就）
- 用编号 → 远端 API 不提供稳定编号，本地编号为自动生成

**理由**：远端成就数据中描述的末尾标点可能与本地不一致（如远端有句号而本地无），去标点后比对可避免误判为不同成就。开发过程中发现这是导致本地比远端多一条数据的原因之一。

## 风险 / 权衡

**[每次爬取都会发送 API 请求]** → 由于无法仅获取元数据，即使缓存有效也需要一次完整请求。但原有逻辑在无缓存时也是一次请求，有缓存时零请求；新逻辑变为始终一次请求。考虑到用户主动点击爬取的频率不高，这是可接受的权衡。

**[缓存文件结构变更]** → 新增 `_cache_meta` 字段。旧版本缓存文件不包含此字段，通过兼容逻辑（视为过期）处理，不会导致崩溃。

**[API 字段依赖]** → 依赖 `lastUpdateTime` 字段的存在和格式。若 API 变更移除该字段，需降级为始终拉取。可通过 `get()` 默认值和异常处理保证健壮性。

**[启动同步的线程安全]** → 后台同步线程完成后通过 Qt 信号回到主线程执行数据合并和 UI 刷新，避免跨线程访问 UI 组件。但如果用户在同步完成前手动操作成就管理页面，可能存在竞争。当前通过 5 秒延迟和静默执行降低了该风险。

**[远端成就被移除的风险]** → 启动同步会自动移除本地存在但远端已删除的成就。如果远端 API 临时异常返回不完整数据，可能导致误删。当前通过要求远端数据量大于 0 来做基本防护。
