## 新增需求

### 需求:缓存元数据记录

系统在保存 API 响应缓存时，必须同时记录 `_cache_meta` 元数据对象，包含 `lastUpdateTime`（来自 API 响应的远端更新时间）和 `cached_at`（本地缓存写入时间）。元数据必须与缓存数据存储在同一个 JSON 文件中。

#### 场景:首次爬取写入元数据

- **当** 用户首次爬取且无缓存文件存在
- **那么** 系统从 API 响应中提取 `lastUpdateTime` 字段，连同当前时间戳一起写入 `_cache_meta` 对象保存到 `achievement_cache.json`

#### 场景:缓存刷新后更新元数据

- **当** 检测到远端数据更新并重新拉取 API 数据
- **那么** 系统必须用新的 `lastUpdateTime` 和当前时间戳覆盖 `_cache_meta`

#### 场景:旧缓存文件兼容

- **当** 缓存文件存在但缺少 `_cache_meta` 字段（历史版本生成的缓存）
- **那么** 系统必须视为缓存已过期，重新请求 API 数据

---

### 需求:远端更新时间获取

系统在爬取前必须向 API 发送请求并从响应中解析 `lastUpdateTime` 字段，用于与本地缓存的 `_cache_meta.lastUpdateTime` 进行比对。该请求必须使用与完整爬取相同的 endpoint 和认证信息。

#### 场景:成功获取远端更新时间

- **当** 用户点击"开始爬取"且认证信息有效
- **那么** 系统必须发送 API 请求，从响应的 `data` 层解析 `lastUpdateTime` 字段并返回

#### 场景:网络请求失败时降级

- **当** 获取远端更新时间的请求因网络错误而失败
- **那么** 系统必须跳过新鲜度检查，回退到使用本地缓存数据（若存在），并通过进度信号通知用户"网络请求失败，使用本地缓存"

#### 场景:认证信息缺失

- **当** 用户未配置 `devcode` 或 `token`
- **那么** 系统必须跳过远端检查，直接使用本地缓存（若存在），否则提示用户配置认证信息

---

### 需求:缓存新鲜度比对

系统在获取到远端 `lastUpdateTime` 后，必须与本地 `_cache_meta.lastUpdateTime` 进行日期字符串比对，决定是否需要重新拉取完整数据。

#### 场景:缓存数据是最新的

- **当** 远端 `lastUpdateTime` 等于本地 `_cache_meta.lastUpdateTime`
- **那么** 系统必须直接使用本地缓存数据，并通过进度信号通知用户"数据已是最新，使用本地缓存"

#### 场景:远端数据已更新

- **当** 远端 `lastUpdateTime` 晚于本地 `_cache_meta.lastUpdateTime`
- **那么** 系统必须重新拉取完整 API 数据，更新缓存文件和 `_cache_meta`，并通过进度信号通知用户"检测到数据更新，正在刷新缓存..."

#### 场景:本地无缓存

- **当** `achievement_cache.json` 文件不存在
- **那么** 系统必须直接请求完整 API 数据并保存缓存（含元数据）

---

### 需求:用户反馈

系统在缓存新鲜度检查的各阶段必须通过 `progress` 信号向用户反馈当前状态，禁止在无提示的情况下静默执行网络请求或使用缓存。

#### 场景:检查更新中

- **当** 系统开始发送远端更新时间检查请求
- **那么** 必须发出进度信号"正在检查数据更新..."

#### 场景:使用缓存

- **当** 比对结果为缓存数据是最新的
- **那么** 必须发出进度信号"数据已是最新（远端更新时间: {lastUpdateTime}），使用本地缓存"

#### 场景:刷新缓存

- **当** 比对结果为远端数据已更新
- **那么** 必须发出进度信号"检测到数据更新（{oldTime} → {newTime}），正在重新获取..."

---

### 需求:启动自动同步

系统在主窗口启动后必须在后台自动检查远端成就数据，与本地 `base_achievements.json` 对比，自动合并新增成就并移除远端已删除的成就，整个过程对用户静默执行，不阻塞 UI。

#### 场景:启动后延迟触发同步

- **当** 主窗口加载完成
- **那么** 系统必须延迟 5 秒后在后台线程发起数据同步检查，不阻塞用户操作

#### 场景:检测到新增成就

- **当** 远端成就数据中存在本地 `base_achievements.json` 中没有的成就（按 `(名称, 描述)` 去末尾标点后作为唯一键比对）
- **那么** 系统必须将新增成就合并到本地数据库，自动检测并注册新分类，重新编号后持久化到 `base_achievements.json`

#### 场景:检测到远端已删除成就

- **当** 本地存在的成就在远端已不存在
- **那么** 系统必须从本地数据库中移除该成就，并在持久化后重新编码所有用户存档以保持一致

#### 场景:无变化

- **当** 远端成就与本地完全一致
- **那么** 系统不做任何修改，静默结束检查

#### 场景:认证信息缺失

- **当** 用户未配置 `devcode` 或 `token`
- **那么** 系统静默跳过启动同步，不显示错误提示

---

### 需求:数据持久化顺序

系统在合并成就数据后，必须严格按照以下顺序执行持久化操作，以避免用户存档与成就数据库不一致。

#### 场景:正确的持久化顺序

- **当** 成就数据合并完成（含新增和/或移除）
- **那么** 系统必须先调用 `save_to_json()` 将合并后的成就写入 `base_achievements.json`，再调用 `reencode_all_user_progress()` 重新编码用户存档，最后调用 `load_local_data()` 刷新 UI

#### 场景:持久化前先编码（错误路径）

- **当** 系统在调用 `save_to_json()` 之前调用 `reencode_all_user_progress()`
- **那么** `reencode_all_user_progress()` 从文件加载的仍是旧数据，导致用户存档编码错误，新成就状态丢失

---

### 需求:内容级缓存校验

当远端 `lastUpdateTime` 与本地一致时，系统必须进一步通过数据内容 hash 比对确认缓存是否真正有效，防止远端时间戳未变但内容已更新的情况。

#### 场景:时间戳相同但内容变更

- **当** 远端 `lastUpdateTime` 等于本地 `_cache_meta.lastUpdateTime`，但 API 响应数据的 MD5 hash 与 `_cache_meta.data_hash` 不同
- **那么** 系统必须用新响应覆盖缓存，并通过进度信号通知"检测到数据内容变更，正在更新本地缓存..."

#### 场景:时间戳和内容均未变

- **当** 远端 `lastUpdateTime` 等于本地，且 MD5 hash 也一致
- **那么** 系统使用本地缓存，通知"数据已是最新"
