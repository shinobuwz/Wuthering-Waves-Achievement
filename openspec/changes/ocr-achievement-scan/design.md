## 上下文

鸣潮成就管理器是一个 PySide6 桌面应用（Python 3.8+，Windows），现有三个 Tab 页（成就管理、统计图表、数据爬取）。成就数据存储在 `base_achievements.json`（约 1000+ 条），用户进度存储在 `user_progress_{uid}.json`，通过 `config.save_user_progress()` 读写。

本地 OCR 引擎 onnxocr（基于 PaddleOCR 转 ONNX）已存在于 `D:\github\onnxocr`，需拷贝到项目目录作为内部模块。

鸣潮成就界面为三级结构：一级分类（左侧图标 Tab）→ 二级子分类（左侧列表）→ 具体成就（右侧卡片列表，可滚动）。固定适配 1920x1080 分辨率。

## 目标 / 非目标

**目标：**
- 检测并定位鸣潮游戏窗口，获取窗口位置和大小
- 截图游戏界面，裁剪出成就列表区域
- OCR 识别成就名称，模糊匹配到数据库
- 判断每条成就的完成状态（"进行中" vs 日期）
- 自动滚动右侧成就列表，扫描当前子分类下的所有成就
- 提供 PySide6 UI 界面集成到现有应用
- 扫描结果写入现有 user_progress 数据结构

**非目标：**
- 不支持后台截图（仅前台窗口）
- 不自动切换一级/二级分类 Tab（用户手动切换，工具只扫描当前页）
- 不适配非 1920x1080 分辨率
- 不修改现有数据结构或 API

## 决策

### 1. 截图方案：mss 前台截图

**选择**: mss 库 + 窗口坐标裁剪
**替代方案**:
- win32gui PrintWindow：支持后台但鸣潮可能阻止
- dxcam：高性能但额外依赖重

**理由**: mss 轻量、跨平台、对前台窗口截图可靠。前台截图已满足需求，无需引入后台截图的复杂性。

### 2. 窗口检测：ctypes + Win32 API

**选择**: 使用 ctypes 调用 FindWindow/GetWindowRect
**替代方案**:
- pywin32：功能更全但依赖更重
- 手动输入窗口位置：不够自动化

**理由**: ctypes 是 Python 标准库的一部分，无需额外安装。只需 FindWindow 和 GetWindowRect 两个 API，足够简单。

### 3. OCR 引擎：onnxocr 内部集成

**选择**: 将 onnxocr 目录拷贝到项目根目录，作为内部模块直接 import
**替代方案**:
- pip install 方式：onnxocr 未发布到 PyPI
- submodule：增加 git 复杂度

**理由**: 直接拷贝最简单，模型文件和代码一起管理，用户无需额外安装步骤。

### 4. 成就条目分割：固定高度像素切割

**选择**: 在 1920x1080 下，按固定像素偏移量切割每条成就卡片
**替代方案**:
- OpenCV 边缘检测：自适应但复杂
- 模板匹配：准确但性能差

**理由**: 成就条目高度固定，1080p 下像素位置可预测。固定切割最简单可靠，无需复杂的图像分析。

### 5. 名称匹配：difflib 模糊匹配

**选择**: `difflib.SequenceMatcher` 计算相似度，阈值 0.7
**替代方案**:
- 精确匹配：OCR 错字会导致匹配失败
- 编辑距离 (Levenshtein)：需要额外库

**理由**: difflib 是标准库，模糊匹配能容忍 OCR 的少量错误。0.7 阈值在容错和准确之间平衡，可后续调优。

### 6. 完成状态判断：OCR 右侧文字

**选择**: OCR 识别成就条目右侧文字，"进行中" = 未完成，日期格式 (YYYY/MM/DD) = 已完成
**替代方案**:
- 图标颜色检测：受亮度/主题影响
- 背景色检测：需要精确的颜色阈值

**理由**: 文字判断最明确，"进行中"和日期差异显著，OCR 识别率高。且 OCR 引擎已加载，几乎零额外成本。

### 7. 滚动去重：成就名称集合对比

**选择**: 每次截图后记录识别到的成就名称集合，如果新一轮与上一轮完全重复则判定到底
**替代方案**:
- 滚动条位置检测：像素级检测不可靠
- 固定滚动次数：不同子分类成就数量不同

**理由**: 名称去重自适应任意长度的成就列表，且逻辑简单。

## 风险 / 权衡

- **[OCR 精度]** 鸣潮使用风格化字体，可能影响识别率 → 缓解：图像预处理（二值化、对比度增强）+ 模糊匹配容错
- **[像素坐标硬编码]** 游戏 UI 更新可能导致坐标失效 → 缓解：将所有坐标常量集中定义，便于调整
- **[滚动动画延迟]** 滚动后需等待动画完成才能截图 → 缓解：固定等待时间（0.5~1s），可配置
- **[模型文件体积]** onnxocr 含模型文件约 50MB+ → 缓解：仅首次需要，后续无额外下载
- **[窗口标题不确定]** 鸣潮窗口标题可能因版本/语言变化 → 缓解：支持多种标题匹配模式，或让用户手动选择窗口

## 已解决的问题

### BUG: 窗口检测匹配到错误窗口

**现象**: FindWindow 按标题匹配，检测到 160x28 的窗口（实际是 VS Code 标题含 "Wuthering-Waves-Achievement"）

**根因**:
1. 鸣潮游戏主窗口由子进程 `Client-Win64-Shipping.exe`（UnrealWindow 类）创建，而非 `Wuthering Waves.exe`（后者仅为启动器，无可见窗口）
2. 窗口标题 `鸣潮` 带尾部空格，`FindWindowW` 精确匹配失败
3. 回退到模糊匹配后，优先匹配到了 VS Code 等含 "Wuthering" 关键词的窗口

**修复**: 改为通过进程名查找 → 获取 PID → 枚举窗口匹配 PID，优先查找 `Client-Win64-Shipping.exe`，并过滤客户区 >= 800x600 排除小窗口

### BUG: 成就列表像素坐标偏移导致 OCR 识别错误

**现象**: OCR 识别出描述文字而非成就标题，状态区域裁到空白

**根因**:
1. 成就列表区域左边界实际在 x=680（有白色分隔线），而非初始估计的 x=540
2. 第一条卡片起始 y=282，而非初始估计的 y=200
3. 卡片周期 194px（卡片 172px + 间隔 22px），而非初始的 148px
4. 名称文字在卡片内的 y=14~44、x=195~620 位置，与初始估计差距大
5. 日期/状态文字在卡片内 y=58~92（中部偏下），而非顶部

**调试方法**: 通过像素亮度扫描（x/y 轴逐像素采样）精确定位卡片边界和文字区域，而非目测估算

**修复**: 更新所有坐标常量：
- LIST: LEFT=680, TOP=282, RIGHT=1845, BOTTOM=1060
- ITEM_HEIGHT=194, CARD_HEIGHT=172
- NAME: LEFT=195, TOP=14, RIGHT=620, BOTTOM=44
- STATUS: LEFT=950, TOP=58, RIGHT=1145, BOTTOM=92
